Overview

Our main task is to develop a voice banking app using text-to-speech (TTS) technology, with a connected frontend and backend, along with optical character recognition functionality. The backend has to be complied into a executable.
Therefore, research was done on 5 components:

  • Backend Language: Python vs Java
  • Frontend Framework: MFC vs .NET
  • TTS Engine: Coqui TTS vs Mary TTS
  • OCR Engine: Tesseract vs EasyOCR
  • Backend Compiler: Nuitka vs PyInstaller vs Py2Exe vs PyOxidizer

To pick a technology for further development, we considered different points of comparison for each section. We will discover, evaluate and analyse the benefits and drawbacks of using different technologies and the reasons behind our final choices.

Backend Language

Points of comparison:

Existing Codebase / Development Time Cost

This is the most important factor, as we are working with Anima v2's backend codebase which was written in Python.
Choosing Java will involve re-writing the entire backend, including finding alternatives to every library that is used. We decided that the time needed for this change is too much.

Performance / Scale of Software

Anima v3 is designed to be small scale which runs locally on the user's device. A main reason to choose Java over Python is for its performance, Java is a compliled language with static typing, Python is a interpreted language with dynamic typing. This means Java is faster as the bytecode is transformed into native machine code before execution [1].
However, we decided that this boost in performance is not neccessary, as we are working with small scale software. We don't need to consider scaling the software to large scales, which is where this performance boost is needed [2].

Team Expertise / Development Time Cost

Python aims to provide clean and concise syntax, making it easier to read and write for developers. In comparison, Java has more verbose syntax, which can lead to longer lines of code [3].
Even though we are proficient in both languages [4], we are more comfortable with continuing developing in Python than in Java.

Summary:

Our choice: Python

Although Java is a faster language, we chose Python since there is already a solid backend codebase in Python, and there is no need to re-write the entire backend for the higher performance.

Frontend Framework

Points of comparison:

Client Request

This is the most important factor, as satisfying our clients' requirements is our top priority.
We had a meeting with our primary contact, Prof. Mohamedally in November 2023. He instructed us to work on Anima v3 using the .NET framework as an upgrade from Anima v2 which uses the MFC framework [5].

Software Timeliness

MFC is a C++ wrapper for Win32 developed in 1992. MFC still gets minor updates today from Microsoft [6]. MFC lacks native support for modern UI frameworks like WPF and advanced communication protocols like WCF [7].
In comparison, .NET framework provides built-in support for modern technologies like WPF (Windows Presentation Foundation), WCF (Windows Communication Foundation), and ASP.NET for web development [7].

Aesthetic Appeal

One of our key requirements is to design a clean, simple and good-looking user interface. The following is more of a subjective analysis, but we believe this is justified since subjective opinions plays a important part for this key requirement.

Summary:

Our choice: .NET

This was an easy decision due to our clients' requirements. However we also did our own research to further justify this choice, which was to choose the more up-to-date and visually-appealing framework - .NET.

TTS Engine

Points of comparison:

Complexity of Setting Up and Usage

This argument is made after we have confirmed our choice of not re-writing the entire backend in Java and keep using Python.
MaryTTS requires manual compilation, server startup, and HTTP query handling for synthesis [11]. In contrast, Coqui TTS provides simpler installation options from PyPI or local cloning, with system dependencies handled via a Makefile [12]. Overall, Coqui TTS stands out for its straightforward setup process, making it the preferred choice for users seeking simplicity in text-to-speech system setup.
MaryTTS is written in Java [13]. The py-marytts package provides a Python interface for MaryTTS. While this is a convenient package, it still requires the MaryTTS server to be running separately [14]. This would make maintenance and deployment harder. Coqui TTS on the other hand is already written in Python and has a Python API where we can run a multi-language model on [15].

Existing Codebase / Development Time Cost

Anima v2 was already using Coqui TTS for their backend, as we are working on their codebase. Changing the TTS engine requires much less re-factoring of code than the previous consideration of changing the language altogether. However, it's crucial to weigh any additional time required for changes against the potential benefits they bring.
At the end, we decided that it was not necessary to switch to Mary TTS and we would continue building upon Anima v2's implementation of Coqui TTS

Summary:

Our choice: Coqui TTS

We have chosen Coqui TTS since it offers a simpler setup process compared to MaryTTS, with straightforward installation options and system dependencies handled efficiently. Additionally, its Python-based implementation aligns well with our existing codebase, reducing the need for extensive refactoring and hosting a HTTP server.

OCR Engine

Points of comparison:

Character Recognition Accuracy

EasyOCR and Tesseract are comparable in character recognition accuracy. However, EasyOCR performs better on scanned files such as PDFs [16], which matches the functionality of Anima v3.

Character Recognition Speed

Assuming both engines are running on CPU only, as we cannot assume our users to have powergul GPUs, then EasyOCR is faster [17].

Speed is an important factor in Anima v3's OCR functionality as we would like to simulate real-time speech of our users.

Ease of Use

Both libraries are very easy to use in Python. Only one line of pip install is needed to start using their Python APIs [18] [19]. Both libraries are very good choices.

Summary:

Our choice: EasyOCR

The differences between EasyOCR and Tesseract in quality is minimal, but EasyOCR performs better with PDFs and is faster in general.

Backend Compiler

Points of comparison:

Compliation Outcome

This is the most important and only factor, as it determines whether the complier works for Anima v3 or not.
We have done extensive efforts and reserach in trying each of the compilers, so we have dedicated sections for our findings:

  • Nuitka
  • PyInstaller and Py2Exe
  • PyOxidizer

Summary:

Our choice: PyOxidizer

PyOxidizer is the only complier that was successful in compliing the Python backend to an executable, as there were issues that were either beyond our expertise, or are not feasible to be resolved. Please check our individual pages for more information.
