Testing Strategy

For our voicebanking solution, ensuring robust and reliable performance through rigorous testing is paramount. We also highly value user feedback and will be conducting user testing to ensure that our product meets the needs of our target users. Our testing strategy will encompass multiple facets to guarantee a seamless user experience and robust functionality. These include:

Unit and Integration Testing
Compatability Testing
Stress Testing
User Testing

Unit and Integration Testing

Unit and integration testing ensures that every individual functionality of AnimaV3 is functioning as intended and are working together successfully.

How was this test conducted?

This test is done through a python script written for pytest. AnimaV3 works by changing the content of the frontend.json file, hence the test is conducted by running the test script and the backend simultaneously and ensuring the correct success message is returned on backend.json.

Compatability Testing

Compatability testing informs the users of the platforms suitable to run AnimaV3. As AnimaV3 is a Windows desktop application build with .NET framework (WPF - Windows Presentation Foundation), it is only compatible with Windows OS. Hence, different versions of Windows OS were tested to ensure compatability.

How was this test conducted?

Used virtual machine to simulate different version of windows system. We chose Windows XP, Windows 7, Windows 8.1, Windows 10 and Windows 11 as the target test environment. Original system images (iso) were downloaded from the official Microsoft website (except for Windows 11 as we already had access to it). Since the support of Windows XP and Windows 7 has been ended, these two system images were downloaded from third-party archive websites. All system images used are original to simulate new installed system environment, making sure that no extra library is already in the system. Usually, if a library is not installed but required to run the software, the relevant debug message will be told to install it. By following these system messages, required libraries were figured out in different system

Example Environment - Windows 8

The results are as follows:

Windows XP: Not Supported
Windows 7: Not Supported
Windows 8.1: Supported, Requirements: Microsoft Visual C++ Redistributable and .NET Framework 4.7.2
Windows 10: Supported, Requirements: Microsoft Visual C++ Redistributable
Windows 11: Supported, Requirements: Microsoft Visual C++ Redistributable

Performance/Stress Testing

Stress testing is required to know the bounds of our application. We need to identify the inputs that don't work with our application so that we can inform the users about it. We also need to know performance of the application under different loads. Hence we are testing for these conditions here.

How was the test for checking input types conducted?

In the Text-to-Speech section, a commonly used 104-key keyboard is used to test text input. Each key on the keyboard was typed into textfield, and then TTS was run by clicking the SPEAK button to see if there is any unsupported character error on backend output.

How was the test for checking peformance under different loads conducted?

In the Text-to-Speech section, texts with varying number of words were tested to see how long it takes for AnimaV3 to create speech. To increase the length of text, the same sentence was repeated multiple times. The sentece being "The sun sets, painting the sky in hues of orange.' was duplicated for increasing the word count." The test was conducted by running the TTS for the same sentence with increasing word count and recording the time taken for the TTS to complete.

Stress Test Demonstration with Different Lengths of Text

The results are as follows:

As seen rom the graph, the cost of converting text to speech increases almost linearly as the number of words that needs to be spoken increases. This is expected as the time taken to convert text to speech should be directly proportional to the number of words in the text.
